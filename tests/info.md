# Authentication, Security, and Testing

One of the areas I spent the most time designing was authentication and security because CaseCompass isn't a typical consumer application. It serves two distinct audiences: individual users and correctional institutions. Individual users can create an account using traditional authentication methods, but institution-managed users, such as incarcerated individuals, often don't have access to email, mobile phones, or verification codes. Because of that, I designed a completely different authentication workflow specifically for correctional environments. Instead of requiring inmates to register themselves, only authorized correctional staff can create accounts through a secure institutional dashboard. Staff members assign a unique username, institution, facility, and role, and the system generates a temporary password that is displayed only once. That password is then provided to the inmate through the institution's approved internal process without relying on email or SMS.

The first time an inmate signs in, they are immediately redirected to a mandatory password change screen. They cannot access the dashboard, AI tools, saved research, or any protected part of the application until that password has been changed. Once they create a new password, the temporary password is permanently invalidated, the account is marked as active, and the user is taken into the Get Started onboarding experience. This ensures that even though institutional staff create the accounts, only the inmate ultimately knows their permanent credentials. If an inmate forgets their password, there is no traditional "Forgot Password" email workflow. Instead, authorized staff generate a new temporary password through the institutional dashboard, and the user is once again required to complete the first-login password change process.

To provide enterprise-level authentication security, I chose **Clerk** rather than building my own authentication system. Clerk is responsible for password hashing, secure session management, authentication, cookies, and credential security using industry best practices. Rather than reinventing authentication, I wanted to rely on a platform whose primary focus is identity and access management. This allowed me to focus my engineering effort on the institutional workflow, authorization model, AI functionality, and legal research experience that make CaseCompass unique.

While Clerk answers the question, **"Who is this user?"**, my **PostgreSQL database hosted on Supabase** answers **"What is this user allowed to do?"** Using **Prisma ORM**, I manage all application-specific data including institutions, facilities, user profiles, roles, permissions, intake sessions, research roadmaps, and audit logs. Separating authentication from authorization creates a cleaner, more scalable, and significantly more secure architecture because identity and application permissions remain independent concerns.

Security extends well beyond the login screen. Every protected request is validated on the server rather than relying on client-side checks. The browser can be manipulated, so I never trust data coming from the client. Every request verifies that the user has a valid authenticated session, belongs to the correct institution and facility, has the appropriate role, has an active account, and has completed the required first-login password change. Even if someone manually enters the URL for a protected page, the server performs authorization checks before returning any data. This server-first authorization model prevents users from bypassing permissions simply by manipulating client-side code.

Because institutional environments often use shared computers, I also designed the application with shared-device security in mind. Sessions are securely managed, local application state is cleared during logout, and sensitive information such as passwords, tokens, and legal narratives are never stored in browser storage when institutional policies disable persistence. This reduces the risk of one user accidentally accessing another user's research on a shared workstation.

Another major focus of this project was building security through **Test-Driven Development (TDD)**. Before implementing any authentication or authorization feature, I first wrote automated tests describing the expected behavior. Those tests intentionally failed at first, confirming they were actually testing the functionality I intended to build. I then implemented only the minimum amount of code necessary to satisfy those tests before refactoring and rerunning the full security test suite. Following the Red-Green-Refactor methodology ensured that security behavior remained intentional, measurable, and protected from future regressions.

The automated test suite validates critical behaviors throughout the application. Tests verify that users cannot bypass the mandatory first-login password change, institution administrators cannot access users outside of their assigned institution, temporary passwords become permanently invalid after use, deactivated accounts cannot authenticate, authorization is always enforced on the server, AI-generated responses conform to a strict schema before reaching the user, and sensitive information such as passwords, authentication tokens, and legal narratives never appear in logs or API responses. This provides confidence that future code changes cannot unintentionally weaken the application's security model.

To improve maintainability, I created a dedicated **Behavior Documentation** system. Every major workflow—including authentication, institutional account creation, authorization, shared-device privacy, AI roadmap generation, and audit logging—is documented in its own Markdown file. Each document explains the expected behavior, the reasoning behind the security decision, which automated tests verify that behavior, implementation details, and any known limitations. This creates living documentation that evolves alongside the application and helps future developers understand not only *what* the system does, but *why* those decisions were made.

The AI functionality was designed with the same security-first mindset. CaseCompass is an educational legal research assistant—not a legal advisor. The AI is intentionally prevented from providing legal advice, predicting court outcomes, recommending legal strategies, calculating filing deadlines, or fabricating legal authorities. Instead, it generates structured legal research roadmaps designed to help users understand legal concepts and identify topics they may wish to research further. Every AI response is validated against a predefined schema before it is displayed. If the generated output fails validation, the response is rejected rather than presented to the user. This layered validation helps ensure consistency, reliability, and factual integrity while clearly communicating the limitations of AI-generated content.
