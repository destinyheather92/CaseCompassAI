# Security Invariants

These rules must always hold. Each one is mapped to the code that enforces it and the test(s) that prove it. If you find yourself changing code in a way that would violate one of these, stop — either the invariant needs an explicit, reviewed update to this document, or the change is wrong.

| # | Invariant | Enforced by | Tested by |
|---|---|---|---|
| 1 | Passwords are never stored in plaintext | Clerk owns all password storage/hashing; Prisma `User` has no password field at all (`prisma/schema.prisma`) | Schema review — there is no column to store one in |
| 2 | Passwords are never written to logs | `lib/security/safe-logger.ts` redacts `password`/`passwordHash`/etc. before any `console.*` call | `tests/unit/security/safe-logger.test.ts` |
| 3 | Temporary passwords are displayed only at creation or reset | `POST /api/institution/users` and `POST /api/institution/users/[id]/reset-password` return the plaintext temp password once, in the response body only — never persisted | *(pending — institutional account creation phase)* |
| 4 | Temporary passwords cannot be retrieved later | No Prisma column stores it; Clerk's password hash is not readable via any API this app calls | *(pending)* |
| 5 | Temporary passwords become invalid after a successful password change | Clerk `user.updatePassword` overwrites the credential | *(pending — first-login phase)* |
| 6 | A user with `mustChangePassword = true` cannot access protected product routes | `lib/auth/authorization.ts` → `requirePasswordChangeComplete`, called by `authorize()`/`requireAuthenticatedUser()` before any role/scope check | `tests/unit/auth/authorization.test.ts`, `tests/integration/authorization/load-app-user.test.ts` |
| 7 | Client-side route hiding is not considered authorization | `proxy.ts` is a coarse UX gate only; every protected server component/route handler independently calls `lib/auth/authorization.ts` against Prisma | *(pending — route wiring phase)* |
| 8 | Authorization is enforced on the server | Same as above — `authorize()` always re-derives role/status/scope from Prisma via the Clerk-verified `userId`, never from client input | `tests/integration/authorization/load-app-user.test.ts` |
| 9 | Institutional administrators cannot access users outside their institution | `requireInstitutionAccess` | `tests/unit/auth/authorization.test.ts`, `tests/integration/authorization/load-app-user.test.ts` |
| 10 | Facility-level administrators cannot access users outside their permitted facilities | `requireFacilityAccess` | `tests/unit/auth/authorization.test.ts` |
| 11 | Deactivated users cannot authenticate | `requireActiveAccount` rejects `DISABLED`; institutional account lifecycle also bans the Clerk user directly (defense in depth) | `tests/unit/auth/authorization.test.ts`; Clerk-ban integration test *(pending)* |
| 12 | Locked users cannot authenticate until an approved recovery action occurs | `requireActiveAccount` rejects `LOCKED` | `tests/unit/auth/authorization.test.ts` |
| 13 | Login errors must not reveal whether a username exists | `createRateLimiter` treats every key identically regardless of prior existence; institution-login uses a single generic "username or password is incorrect" message | `tests/unit/security/rate-limit.test.ts` ("does not expose account existence"); login-copy test *(pending)* |
| 14 | Password hashes must never appear in API responses | `AppUser`/Prisma `User` selections never include a password field (none exists to select) | *(pending — API route phase)* |
| 15 | Guest roadmap generation must be rate-limited | `lib/security/rate-limit.ts` reused for `/api/intake/roadmap` | *(pending — roadmap phase)* |
| 16 | Authenticated roadmap generation must also be rate-limited | Same limiter, keyed by user id instead of IP | *(pending)* |
| 17 | Raw case descriptions must not appear in analytics | No analytics integration exists in this build | N/A currently |
| 18 | Raw case descriptions must not be included in ordinary application logs | `lib/security/safe-logger.ts` redacts `description`/`prompt` keys | `tests/unit/security/safe-logger.test.ts` |
| 19 | Guest intake information must not be silently stored on the server | `IntakeSession` rows are only created when a roadmap is actually requested, never on each step | *(pending — intake phase)* |
| 20 | Shared-device logout must clear sensitive client state | Zustand intake store namespaced per user, cleared on logout | *(pending — intake phase)* |
| 21 | Institution-managed accounts must not require email or phone verification | Institutional accounts are created via Clerk's Backend API with only `username`+`password` | *(pending — institutional account creation phase)* |
| 22 | Role checks must occur on every protected server action and API request | `authorize()`/`requireAuthenticatedUser()` is the mandatory entry point for protected routes | ongoing — verified per-route as each is built |
| 23 | Institution and facility scope must come from the authenticated account, not trusted client input | `requireInstitutionAccess`/`requireFacilityAccess` only ever read `user.institutionId`/`user.facilityId` from the Prisma row loaded via the verified Clerk id — no parameter accepts a client-supplied scope override | `tests/integration/authorization/load-app-user.test.ts` ("never trusts a client-supplied role or institutionId") |
| 24 | Roadmap generation must not fabricate legal authorities | Not yet built — see `docs/behavior/roadmap-generation.md` | *(pending — future roadmap-generation phase)* |
| 25 | AI output must be schema-validated before being returned | `lib/intake/intake-interview-schema.ts` (Zod, strict, `.nullable()` not `.optional()` for Structured Outputs strict-mode compatibility); re-validated independently in `OpenAIIntakeInterviewer` even after the SDK's own parse | `tests/unit/intake/intake-interview-schema.test.ts`, `tests/unit/ai/openai-intake-interviewer.test.ts` |
| 26 | Invalid AI output must fail safely | `output_parsed === null` or a failed re-validation both return a safe `invalid-output` result, never partially-valid data | `tests/unit/ai/openai-intake-interviewer.test.ts` |
| 27 | A failed roadmap/intake request must not erase intake answers | An `IntakeAnswer` row is persisted *before* the next AI call; a provider failure never loses the just-given answer, and the Zustand store's `setError` never clears other fields | `tests/integration/intake/submit-intake-answer.test.ts` ("preserves the saved answer..."), `tests/unit/intake/use-intake-store.test.ts` |
| 28 | Resetting a temporary password must invalidate the previous temporary password | Clerk `updateUser({password})` overwrites the credential; Prisma `mustChangePassword` reset to `true` | *(pending)* |
| 29 | Audit logs must not contain secrets or full case narratives | `recordAuditEvent` redacts metadata via the same guard as `safeLog` | `tests/integration/audit/audit-log.test.ts` |
| 30 | A user cannot assign themselves a higher role | Role is only ever set server-side by an `INSTITUTION_ADMIN`/`SYSTEM_ADMIN` acting through `requireRole`-gated endpoints; no endpoint lets a user set their own role | *(pending — institutional account creation phase)* |
| 31 | Institution staff cannot create system administrators | Institution user creation endpoint restricts `role` to `incarcerated-user \| educator \| legal-aid-staff` at the Zod schema level | *(pending)* |
| 32 | Users cannot bypass the first-login password requirement by entering a route directly | Enforced server-side via `requirePasswordChangeComplete` inside every protected server component, not just `proxy.ts` | *(pending — route wiring phase)* |
| 33 | Session expiration must be enforced on the server | Delegated entirely to Clerk, which validates session tokens server-side on every `auth()` call | ongoing |
| 34 | Disabled accounts must lose access even if a prior client state still exists | `requireActiveAccount` is re-checked on every request via `authorize()` — there is no cached "is active" client flag that grants access | `tests/unit/auth/authorization.test.ts` |
| 35 | User-provided text must never be rendered as raw HTML | React's default escaping is relied on everywhere; `dangerouslySetInnerHTML` is not used anywhere in `components/onboarding/` (confirmed by grep during the AI-intake-interview phase, re-checked at final verification) | `tests/components/onboarding/adaptive-question.test.tsx` ("renders user-provided question text safely — never as raw HTML") |
| 36 | The AI intake interviewer never answers legal questions, gives advice, predicts outcomes, or recommends filings | System prompt directives (`lib/ai/prompts/intake-interviewer-system.ts`); behavioral smoke-tested via the "asks for legal advice" evaluation fixture | `tests/unit/ai/intake-interviewer-system-prompt.test.ts`, `tests/integration/intake/evaluation-fixtures.test.ts` |
| 37 | Only one question is returned per AI turn | Structural — `IntakeInterviewResponseSchema.question` is a single nullable object, not an array; a response with an array in that field fails validation | `tests/unit/intake/intake-interview-schema.test.ts` ("rejects a response whose 'question' field is an array") |
| 38 | Users cannot access another user's intake session | `checkIntakeSessionAccess()` — an authenticated user must match `session.userId` exactly; a guest session's cuid id is its own credential | `tests/unit/intake/intake-access.test.ts`, `tests/integration/intake/submit-intake-answer.test.ts`, `get-and-complete-intake-session.test.ts` |
| 39 | AI prompts are constructed server-side only | `lib/ai/prompts/*` are never imported by a `"use client"` file — confirmed by grep at final verification; the client only ever calls `/api/intake/interview/*` and receives structured JSON, never prompt text | grep-verified at final verification pass |
| 40 | OpenAI credentials never reach the browser | `lib/ai/openai-client.ts` is `import "server-only"`; `OPENAI_API_KEY` is read only via `lib/env.ts`, never referenced in any client component | `tests/unit/ai/openai-client.test.ts` |
| 41 | Complete prompts and full user narratives are not logged | Audit metadata for every `intake_interview_*` event contains only counts/statuses/failure reasons, never `factualSummary` or answer text | manual code review of `lib/intake/start-intake-session.ts`/`submit-intake-answer.ts`'s `recordAuditEvent` calls (no test currently asserts this negatively for the intake events specifically — see Known Limitations) |
| 42 | Intake completion always requires explicit user review | `completeIntakeSession` only accepts `acknowledged: true` while `status === "READY_FOR_REVIEW"`; the AI reaching `intake-complete` only produces `ready-for-review`, never `completed` directly | `tests/integration/intake/get-and-complete-intake-session.test.ts` |
| 43 | Roadmap generation is a separate system from intake | No roadmap-generation code exists in this phase; `IntakeSession.status` stops at `completed` — see `docs/behavior/roadmap-generation.md` | N/A — enforced by absence; will need a real test once that phase exists |
| 44 | The maximum AI question count is enforced by the server, not the model | Checked against the actual persisted `IntakeAnswer` row count before any further provider call | `tests/integration/intake/submit-intake-answer.test.ts` ("forces ready-for-review once the server-enforced question limit is reached") |
| 45 | A user can only ever see their own dashboard resources (intakes, roadmaps, saved items) | `requireOwnedIntake`/`requireOwnedRoadmap`/`requireOwnedSavedItem` (`lib/auth/dashboard-authorization.ts`) check `resource.userId === actorUser.id` directly, on every read and write path | `tests/integration/authorization/dashboard-authorization.test.ts`, `tests/integration/dashboard/get-intake-detail.test.ts`, `get-roadmap-detail.test.ts`, `tests/integration/saved/remove-saved-resource.test.ts` |
| 46 | Institution staff get no automatic access to a user's private dashboard research | Ownership checks (#45) are role-blind — there is no `INSTITUTION_ADMIN` bypass anywhere in `lib/auth/dashboard-authorization.ts` or the services that call it | Same as #45 — no test asserts a staff-role bypass exists because no code path grants one |
| 47 | Another user's dashboard resource resolves to `not-found`, never `forbidden` | Ownership-check failures return `{ok:false, reason:"not-found"}`; `app/dashboard/intakes/[intakeId]/page.tsx` and `roadmaps/[roadmapId]/page.tsx` call Next's `notFound()` on that result rather than rendering a distinguishable "access denied" state | `tests/integration/authorization/dashboard-authorization.test.ts`, `tests/integration/dashboard/get-intake-detail.test.ts` ("returns not-found for an intake owned by a different user") |
| 48 | The dashboard timeline never fabricates a date the user didn't provide | `buildIntakeTimeline` only produces an exact date from a strict `YYYY-MM-DD` `date`-type answer; every other timing-relevant answer is labeled "Date not provided" | `tests/unit/dashboard/timeline-mapper.test.ts` |
| 49 | Roadmap generation requires an explicitly confirmed intake, never an AI-reached "complete" state alone | `createRoadmapFromIntake` only proceeds when `intake.status === "COMPLETED"` (set only by `completeIntakeSession`'s explicit-acknowledgement path, invariant #42) | `tests/integration/roadmap/create-roadmap-from-intake.test.ts` ("rejects generation for an intake that hasn't been confirmed yet") |
| 50 | Roadmap progress can never be recorded against a step that doesn't exist in the generated roadmap | `updateRoadmapProgress` looks up `stepId` in the roadmap's own `content.steps` before writing any `RoadmapProgress` row | `tests/integration/roadmap/update-roadmap-progress.test.ts` ("rejects a stepId that doesn't exist on this roadmap's content") |
| 51 | A saved item's `href` cannot be a `javascript:`/`data:` URI or unencrypted `http://` | `saveResourceSchema`'s `SAFE_HREF_PATTERN` only allows relative internal paths or `https://` URLs | `tests/unit/saved/saved-resource-schema.test.ts` ("rejects a javascript: URI href", "rejects a plain http (non-https) external href") |
| 52 | Dashboard activity events are drawn from a fixed allowlist and can never store a full case narrative | `recordUserActivity` no-ops for any `type` not in `APPROVED_ACTIVITY_EVENTS`, and hard-caps `title`/`description` length | `tests/integration/activity/record-user-activity.test.ts` |
| 53 | "Clear My Session" clears local client data and signs the user out even if the server-side audit call fails | `clearAllLocalSessionData()` and Clerk `signOut()` run in the confirm handler's `finally` block, independent of the `fetch` outcome | `tests/components/dashboard/clear-session-button.test.tsx` ("still clears local data and signs out even when the server notification fails") |
| 54 | `/dashboard(.*)` is gated by proxy as a coarse first pass, but every layout/page independently re-derives the user from Prisma | `app/dashboard/layout.tsx` calls `requireDashboardAccess()` itself; not solely reliant on `proxy.ts` matcher coverage | Manual verification per invariant #7/#8's existing pattern; `tests/integration/dashboard/*-route.test.ts` cover the API-route side of the same principle |
| 55 | Case search never fabricates a case — every result comes from a real provider record, never AI memory | `lib/case-search/courtlistener-provider.ts`'s `mapResult` only returns a record with a real `caseName`/`court`/source URL; unconfigured provider or provider failure returns a safe "unavailable" message, never invented results | `tests/unit/case-search/courtlistener-provider.test.ts`, `tests/unit/case-search/case-search-service.test.ts` |
| 56 | A verified case's `jurisdiction` always comes from the roadmap itself, never a client-supplied value | `buildRoadmapCaseRequest` reads `content.jurisdiction.code` unconditionally; the route's `extractOverrides` helper never reads a `jurisdiction` key from the request body | `tests/unit/case-search/build-roadmap-case-request.test.ts` ("ignores a client-supplied jurisdiction override"), `tests/integration/case-search/roadmap-cases-route.test.ts` ("ignores a client-supplied jurisdiction and always uses the roadmap's own") |
| 57 | A case is never claimed to still be good law without a real citator check | `laterHistoryStatus` defaults to `"not-checked"`; the UI always shows the required "has not confirmed whether this decision has been limited..." notice while that status holds | `tests/components/roadmap/case-result-card.test.tsx` |
| 58 | A saved case always carries its underlying verified source record, never just an AI summary | `SavedCase` requires `sourceUrl`/`sourceName`/`providerCaseId` at the schema level (`saveCaseSchema`); `sourceUrl` must be `https://` | `tests/unit/saved/saved-case-schema.test.ts` (implicit via `saveCaseSchema` validation), `tests/integration/saved/save-case.test.ts` |
| 59 | A user can only ever see, save, or delete their own saved cases | `requireOwnedSavedCase` returns `not-found` for another user's saved case; roadmap-scoped case routes require `requireOwnedRoadmap` | `tests/integration/authorization/dashboard-authorization.test.ts` ("requireOwnedSavedCase"), `tests/integration/authorization/cross-role-cross-user-regression.test.ts` |
| 60 | Institution-managed users cannot self-serve credential changes through the dashboard | `app/dashboard/settings/page.tsx` renders Clerk's `<UserProfile>` only for `role === "INDIVIDUAL"`; every other role sees the fixed "managed by authorized institutional staff" messaging instead, decided server-side from the Prisma-derived role, never a client value | Manual verification (role-conditional server component render) |
| 61 | Institution staff routes and API endpoints reject non-admin roles, including institution-managed end users | `requireAuthenticatedUser({roles:["INSTITUTION_ADMIN"]})` on every `/api/institution/*` route | `tests/integration/institution/users-route.test.ts` ("returns 403 for an authenticated user who is not an institution-admin"), `tests/integration/authorization/cross-role-cross-user-regression.test.ts` |

"*(pending)*" entries are invariants whose enforcing code hasn't been built yet in this session — they're listed now so later phases are held to them, and this table is updated (not silently left inaccurate) as each one lands.
